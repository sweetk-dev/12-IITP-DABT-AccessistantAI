# 장애인 지원 정책 DB

장애인등록혜택전단지(2026-03-09 기준)의 약 40개 지원 항목 각각에 대해, **실제 당사자가 신청·이용하는 데 필요한 정보**(근거 법령, 신청처, 필요 서류, 발급 기간, 비용, 유효기간, 환승·예외, FAQ, 문의처, 출처)를 구조화해 모아둔 데이터베이스. AI 비서가 자연어 질문에 답할 때 이 DB를 검색하도록 설계.

## 폴더 구조

```
정책DB/
├── README.md                     ← 이 문서
├── schema.json                   ← 모든 항목 JSON이 따르는 표준 스키마
├── crawl_targets.json            ← 모든 출처 URL과 재크롤링 메타데이터(단일 진입점)
└── items/
    ├── B001_subway_metro_free.json
    ├── B001_subway_metro_free.md
    ├── B002_railroad_discount.json
    ├── B002_railroad_discount.md
    └── ...  (B0xx 항목들)
```

- 각 항목은 **JSON(시스템·AI용)** 과 **Markdown(사람용)** 을 **쌍**으로 생성.
- 항목 ID 체계: `B001` ~ `B043` (전단지 항목 순서, 이후 B044+ 확장).
- `schema.json` 은 JSON 스키마 Draft-07. CI에서 모든 `items/*.json` 을 검증.

## 표준 항목 템플릿 (요약)

모든 항목 JSON은 `schema.json` 을 따른다. 주요 필드:

- `id`, `leaflet_section`, `leaflet_number`, `title`, `short_summary`, `category`, `benefit_type`
- `supported_amount` — 할인율·범위
- `eligibility` — 대상자, 중증/경증, 동반자, 소득·연령·거주지 조건
- `legal_basis` — 법령·시행령·시행규칙·조례
- `operating_agencies` — 운영기관별(지역별) 차이
- `how_to_use` — 카드 보유/미보유, 예외, 환승 처리
- `application` — 신청처(채널·방법·URL), 필요 서류, 처리 기간, 비용, 온라인/대리 가능 여부
- `validity` — 유효기간, 갱신
- `exceptions_and_caveats` — 예외·주의사항
- `penalties_for_misuse` — 부정사용 제재
- `faq` — 자주 묻는 질문(Q/A)
- `contact` — 문의처(전화·URL)
- `sources` — **출처(재크롤링 메타데이터 포함)**
- `related_items` — 연관 항목 ID
- `last_verified`, `version`

## 출처 추적·재크롤링 정책

장애인 정책은 자주 바뀌고 공식 API가 없는 경우가 많아, 모든 출처는 `crawl_targets.json` 에 모아두고 정기 크롤러가 변경을 감시한다.

### 우선순위
- **primary** — 법령(law.go.kr), 보건복지부, 법제처 '찾기쉬운 생활법령정보', 광역지자체 공식 안내.
- **secondary** — 운영기관 공식 페이지·약관 PDF.
- **supplementary** — 보조 안내(복지로 블로그, 복지뱅크 등).

### 권장 크롤 주기
| 출처 유형 | 권장 주기 |
|---|---|
| 운영요금·할인율(별도운임 등 자주 변경) | monthly |
| 법령·시행령 | monthly (API 호출이면) / quarterly (페이지 크롤이면) |
| 정부·지자체 안내 페이지 | quarterly |
| 보조 자료 | quarterly ~ on_demand |

### 변경 감지 방식
- `page_hash` — HTML 본문 텍스트 SHA-256.
- `pdf_hash` — PDF 파일 SHA-256 (+ 파일명 날짜 패턴 감시).
- `last_modified_field` — API 또는 페이지에 노출된 '수정일/시행일' 필드.
- `css_selector_text` — 특정 CSS 셀렉터로 추출한 텍스트만 비교.
- `manual_review` — 자동 감지 곤란, 분기 1회 수동 검토.

### 공식 API 활용
- **국가법령정보 Open API** (law.go.kr/DRF/lawService.do) — 법령 텍스트 변경 모니터링에 사용.
- **공공데이터포털 API** (api.odcloud.kr) — 도시철도 노선·운영기관·교통약자센터 메타데이터.

대부분의 운영기관(서울교통공사·코레일·지역 도시철도공사·민간 운영사)은 정책 본문에 대한 공식 API를 제공하지 않으므로 페이지 해시·PDF 해시 기반 크롤링이 기본.

## 데이터 확장 작업 순서 (다음 항목 작성 시)

1. 새 항목 ID(예: B044) 결정 → 전단지에서 원문 짧은 요약 확보.
2. WebSearch + 국가법령정보·관할 부처·운영기관 공식 페이지 조사.
3. `items/B044_xxx.json` 작성 (`schema.json` 준수).
4. `items/B044_xxx.md` 작성 (위 B001 문서 구조를 그대로 따름).
5. `crawl_targets.json` 의 `targets[]` 에 새 출처 추가 (`used_by_items` 에 B044 추가).
6. 검증: 스키마 검증 + 핵심 사실 교차 확인.
7. 이 README의 표에 ✅ 표시.

## 검증

각 JSON 파일은 다음을 통과해야 한다:
- `schema.json` JSON Schema 검증 (jsonschema 등으로 headless).
- 모든 URL이 https로 시작하거나 명시적으로 http가 허용된 출처(예: 일부 .go.kr 페이지).
- `sources[].priority` 가 최소 1개 이상의 `primary` 를 포함.
